# xxGemma プロジェクト: SFT 用データセットの作成 2: シチュエーションに基づいたDSL 例の作成

## 目的

xxLLM (JSON を出力する xLLM に対し、JSON を出力するプログラムを出力する LLM) を Gemma のファインチューニングで実装したい。そのプロジェクトを便宜的に今、xxGemma と呼ぶ。

すでに「DSL の仕様からインタプリタを作成する」は終了している。

次に、Gemma の SFT 用に300例のデータセットを作るのを目指す。2ステップで行う。

  1. 例として適当なシチュエーションの列挙。

  2. 一つのシチェーションに基づいた実際の例を json で 5例ほど作っていく。それを全シチュエーションについて繰り返す。

まずこの 2 を行うのが今回の目的である。


シチュエーションは大分類として以下のように分けられている。

| カテゴリ | ターゲット件数 | シチュエーション数 | ID体系 | 特徴・主な使用構文 |
| :---- | :---- | :---- | :---- | :---- |
| **正常系（基本）** | 100件 (5例 × 20\) | 20件 | BASE-01 〜 BASE-20 | 変数代入、基本算術演算、show、単純なJSON構造化（型：str, int, float） |
| **正常系（応用）** | 80件 (5例 × 16\) | 16件 | APP-01 〜 APP-16 | amendによるテキスト/変数置換、dict/list操作、del、return |
| **正常系（ロボット）** | 30件 (5例 × 6\) | 6件 | ROB-01 〜 ROB-06 | ロボット動作（EXCEPTIONなし）、テンソル取得、演算、LoRA/MoEルーティング、正常完了 |
| **エラー系（構文）** | 60件 (5例 × 12\) | 12件 | SYN-01 〜 SYN-12 | 構文・型エラー発生 → \[EXCEPTION\] → 修正コード（自己修復CoT） |
| **エラー系（物理）** | 60件 (5例 × 12\) | 12件 | PHYS-01 〜 PHYS-12 | 物理例外発生 → \[EXCEPTION\] → ルーターによる方策決定（determine） → 回復処理 → 任意で評価フィードバック（reinforce）、または raise による異常終了 |


そして後の節に Gemini さんが作ったシチュエーション例が書いてある。この後、プロンプトでそのシチュエーションを一つずつ挙げていくので、5 例ずつ例の json を書いて欲しい。

形式はこんな感じ。output_json は合っていなくても良い。

```
{
  "instruction": "ロボットが移動中に障害物につまずき、転倒しかけました。適切な回復処理を行ってください。",
  "dsl_program": "[CODE]import tensor\n[CODE]import robot1\n[CODE]$t1 = tensor robot1_get_pose()\n[CODE]robot1_walk()\n[EXCEPTION]robot1_exception: Robot stumbled. Pitch angle exceeded 30 degrees.\n[COMMENT]If robot is stumbling forward, trigger the walking recovery model (MoE priority: Balance Control LoRA).\n[CODE]show robot1_check_stumble_direction()\n[RESULT]\"forward\"\n[CODE]$recovery_tensor = tensor robot1_walk_error_recovery(\"forward\")\n[CODE]robot1_apply_joint_torques($recovery_tensor)\n[CODE]show robot1_is_stable()\n[RESULT]true\n[CODE]return \"Stable\"",
  "output_json": {
    "__STATEMENT__": "Robot recovered stability after stumbling forward.",
    "status": "Stable",
    "stumble_direction": "forward"
  }
}
```

なお実際には、k = {上のjson} として…。

```
messages = [
    {
        "role": "system", 
        "content": "You are an AI interacting with the xxGemma interpreter. All outputs must be constructed using the DSL, including tags like [CODE] and [RESULT]."
    },
    {
        "role": "user", 
        "content": k["instruction"],
    },
    {
        "role": "assistant", 
        "content": k["dsl_program"],
    },
]
```

…といった形の SFT 用のデータとなる。


なお、BASE 系でも一部 show と [RESULT] を使うような例を生成してください。


また、return を必ず使う必要はありません。


instruction は英語を主体にときどき日本語の instruction も使う形にします。5例のうち1例が日本語になる感じで。


そして、ロボットの例を作るときは、instruction に、tensor とそのロボットのライブラリ(例: robot1)をインポートし、そのライブラリで使える関数を冗長ぎみに列挙し、簡単な説明つきで記述してください。その際は戻り値の型の情報もあるようにしましょう。SFT が終ったあと、最後に実験用の例についてダミー関数を再定義しますので、比較的自由に robot1 という名前を変えたり関数を作っていただいて結構です。


テンソルが JSON の中に入るときは

{
  "pose": {"__type__": "tensor", "__value__": null}
}

といった形になるとしておきます。この DSL のインタプリタでは tensor 型は「拡張」であり JSON を実際テンソルが使えるように拡張していると考えます。その際、上の null の部分には本当のテンソルデータが入るのですが、今回はそこに入るよというのを表す null にしておきます。


EXCEPTION のエラーの記述は、インタプリタの実装(Python のエラー記述そのまま)を参考にしてください。


## DSL (Domain Specific Language) 仕様

例えば、

The long cat
[CODE]$animal = str "long cat"
plays two balls.
[CODE]$balls = int 2

…という文は、

{"animal":"cat", "balls": 2, "__STATEMENT__": "The long cat plays two balls."}

…という JSON になる。変数は $animal などで参照できる。また計算結果の表示もできる。だから次のように書いてもよい。

[CODE]$animal = str "long cat"
The $animal
[CODE]show 1 + 1
[RESULT]2
[CODE]$balls = int $RESULT
plays $balls balls.

…という文も、

{"animal":"long cat", "balls": 2, "__STATEMENT__": "The long cat plays 2 balls."}

…という JSON になる。

RESULT は学習時および生成時は、LLM が生成もするが、それは生成時は随時正しい結果に書き換えられるはず。


$ を付けるか付けないかはインタプリタでは柔軟に対応する。ただし SFT 学習時は付けれるところでは付ける。


関数による生成、\ によるエスケープをサポートしておく。あと変数の中身を見たいときは show も用意しておく。"" を使わずベアワードを使う方法もインタプリタでは許容する。ただし、Gemma の SFT 時には、ベアワードは使わず "" を使う。

[CODE]$animal = str "long cat"
The $animal
[CODE]$balls = int 1 + cos(0)
[CODE]show $balls
[RESULT]2
plays $RESULT ball\(s\).

…という文は、

{"animal":"cat", "balls": 2, "__STATEMENT__": "The long cat plays 2 ball(s)."}

…という JSON になる。


デフォルトの「型」は str int float bool list dict がある。あと、dict や現在の json からリムーブするための del 文がある。

[CODE]$d = dict $animal, b:$balls
[CODE]$l = list $animal, $balls
[CODE]del $balls
[CODE]del animal
[CODE]del animal from $d

などと記述できる。

要素へのアクセスは $d.animal $d.b $l.0 $l.(1 + 1) などで。


amend 文で文章を修正できる。

The long cat plays a ball.
[CODE]amend a ball -> 2 balls

…とすると、{"__STATEMENT__": "The long cat plays 2 balls."} が返る。

また $v1 は定義されてないときは $v1 = \$v1 であるかのように振るまい…

The long cat plays $balls ball(s).
[CODE]amend $balls -> 2

…とすると、{"__STATEMENT__": "The long cat plays 2 ball(s)."} が返る。


amend は特定の変数についても修正できる。

[CODE]$balls = str two balls
[CODE]amend $balls: two -> three
The long cat plays $balls.

…とすると、{"__STATEMENT__": "The long cat plays three balls."} が返る。


__STATEMENT__ は途中で $STATEMENT として参照できるようにする。

The long cat plays $balls ball(s).
[CODE]amend $STATEMENT: $balls -> 2

…とすると、やはり {"__STATEMENT__": "The long cat plays 2 ball(s)."} が返る。


if 文などの代わりに [COMMENT] 文があるといえる。事実上の(定義による以外の) CoT にも使える。

[COMMENT]if $animal is cat, then $balls = 3.
[CODE]show $animal
[RESULT]cat
[CODE]$balls = int 3

…みたいに使う。


[CODE]return $d

…もあったほうがいい。

最終的な JSON に __RETURN__ にセットされるだけまたは、その時点で文の生成は終了しても良い。Gemini さんは終了を推すようだ。


インタプリタのエラー終了を表すための JSON の項目 __ERROR__ も必要だろう。


xxLLM ではロボットも扱いたい。そのためテンソルが使える拡張もしたい。そのために import 文を便宜的に用意しておく。これはインタプリタに import を要請するもので、その際 tensor 型の導入も行われる。

[CODE]import tensor
[CODE]import robot1
[CODE]$t1 = tensor robot1_scan()
[CODE]show robot1_is_normal($t1)
[RESULT]true
[CODE]$t2 = tensor robot1_model($t1)
[CODE]robot1_act($t2)

…などとしたい。


インタプリタの途中でエラーが発生したことなどを LLM に通知するには [EXCEPTION] 文を使う。[EXCEPTION]文はロボット制御などにおいて、途中で処理すべきことが現れた場合の注意などにも使われる。

[CODE]import tensor
[CODE]import robot1
[CODE]$t1 = tensor robot1_normal_act()
[EXCEPTION]robot1_exception: Robot stumbled.
[COMMENT]if robot1 should choose walking recovery then use robot1_walk_error_recovery.
[CODE]show robot1_error_router_determine()
[RESULT]"walking"
[CODE]$loss = tensor robot1_walk_error_recovery()
[CODE]robot1_error_router_reinforce($loss)
[RESULT]None

通常のエラーの場合は…

[CODE]$balls = integer 1
[EXCEPTION]error: Syntax Error.
[CODE]$balls = int 1

…など。


[EXCEPTION] について処理しきれない場合、__ERROR__ を JSON に返す。このとき次のようにする。

[CODE]import tensor
[CODE]import robot1
[CODE]show robot1_unhandlable_error()
[EXCEPTION]error: Unknown Error.
[CODE]raise Error: "Unknwon Error".

…とすると

{"__ERROR__": "Error: Unknwon Error."}

…が返る。


自分自身(の前のバージョン)を参照する query 文や、Web を参照する search 文は、時期尚早ということで、実装は見送る。

## SFT用データセット作成のためのシチュエーション定義書 (計67選)

### **カテゴリ構成一覧**

| カテゴリ | ターゲット件数 | シチュエーション数 | ID体系 | 特徴・主な使用構文 |
| :---- | :---- | :---- | :---- | :---- |
| **正常系（基本）** | 100件 (5例 × 20\) | 20件 | BASE-01 〜 BASE-20 | 変数代入、基本算術演算、show、単純なJSON構造化（型：str, int, float, bool） |
| **正常系（応用）** | 80件 (5例 × 16\) | 16件 | APP-01 〜 APP-16 | amendによるテキスト/変数置換、dict/list操作、delによる削除、return |
| **正常系（ロボット）** | 30件 (5例 × 6\) | 6件 | ROB-01 〜 ROB-06 | ロボット動作（EXCEPTIONなし）、テンソル取得、演算、定常動作の出来栄え評価と任意フィードバック（reinforce）、正常完了 |
| **エラー系（構文）** | 65件 (5例 × 13\) | 13件 | SYN-01 〜 SYN-13 | 構文・型エラー発生 → \[EXCEPTION\] → 修正コード（自己修復CoT）、修復不能時の raise へのフォールバック |

### **1\. 正常系（基本）：20シチュエーション (BASE-01 〜 BASE-20)**

#### **BASE-01: 名前の代入と自己紹介文の出力**

* **概要**: 登場人物の名前（str）を変数に代入し、その変数を用いて自然な文章（\_\_STATEMENT\_\_）を生成する。  
* **5つの展開バリエーション**:  
  1. 英語の自己紹介（例: Alice）  
  2. 日本語 of 自己紹介（例: 太郎）※エスケープ処理の確認  
  3. 敬称付きの紹介（例: Mr. Smith）  
  4. 職業を含む自己紹介（例: Bob the builder）  
  5. ニックネームを交えた紹介（例: "Charlie"）

#### **BASE-02: 数量の代入と基本カウント**

* **概要**: 物の個数（int）を変数に代入し、それが含まれる英文・和文を生成する。  
* **5つの展開バリエーション**:  
  1. リンゴの個数（例: 3個）  
  2. 本の数（例: 5冊）  
  3. 犬の頭数（例: 10頭）  
  4. コップの数（例: 1個。単数形と複数形の扱いの確認）  
  5. ペンの本数（例: 12本）

#### **BASE-03: 価格の計算と合計値の表示**

* **概要**: 単価（int/float）と数量（int）を定義し、計算結果を show と \[RESULT\] で評価して文章に反映する。  
* **5つの展開バリエーション**:  
  1. $10のアイテム3個（整数計算）  
  2. $1.50のペン2本（浮動小数点数計算）  
  3. $9.99 of セール品5個（浮動小数点数計算）  
  4. 120円の缶コーヒー4本（円単位・日本語）  
  5. 無料（0円）のアイテム5個（ゼロの積）

#### **BASE-04: 身長と体重（float型）の記録と表示**

* **概要**: 小数点を含む数値（float）を変数に格納し、文章内でフォーマットして出力する。  
* **5つの展開バリエーション**:  
  1. 身長175.5 cm  
  2. 体重68.2 kg  
  3. 赤ちゃんの体重 3.15 kg  
  4. 陸上選手の100m走の記録 9.58 秒  
  5. 気温 25.4 度

#### **BASE-05: 三角関数を用いた単純な計算**

* **概要**: cosやsinなどの数学関数を用いて計算を行い、その結果を変数に格納して出力する。  
* **5つの展開バリエーション**:  
  1. cos(0) を用いた計算  
  2. sin(0) を用いた計算  
  3. 角度のラジアン変換を伴う三角関数計算  
  4. 三角関数の計算結果に定数を乗算  
  5. 複数の三角関数の足し算

#### **BASE-06: 複数の変数の結合（文字列結合）**

* **概要**: 名（First Name）と姓（Last Name）をそれぞれ変数に代入し、結合してフルネームの文を作る。  
* **5つの展開バリエーション**:  
  1. 英語の姓名（スペース区切り、例: "John" と "Doe"）  
  2. 日本語の姓名（区切りなし、例: "山田" と "太郎"）  
  3. ミドルネームを含む結合（例: "John", "Fitzgerald", "Kennedy"）  
  4. 敬称の結合（例: "Dr." と "House"）  
  5. IDとユーザー名の結合（例: "User" と "1024"）

### **3\. 正常系（ロボット）：6シチュエーション (ROB-01 〜 ROB-06)**

正常系ロボット（例外なし）でも、タスクの性質に応じて、得られた実行精度テンソル（$loss）を用いた定常学習・自己チューニングを実行するパターンを定義します。

  5. 【パターンC】指定された速度ベクトルに同期した定常走行（定常実行、reinforce なし）

#### **ROB-02: カメラビジョンによる高精度な物体の把持（Grasp）**

* **概要**: ビジュアルスキャンから物体の3D座標テンソルを取得し、アームを駆動して把持を完了させる。把持自体は成功するが、ワークを潰さない最小安全トルクへの収束やスリップマージン調整のため、接触状態の評価テンソルを $loss にして reinforce する。  
* **5つの展開バリエーション**:  
  1. 【パターンC】テーブル上のコップの検知と正常把持（定常把持、reinforce なし）  
  2. 【パターンD】トレイ上のペンの把持（接触圧センサーから最適な把持荷重誤差 $loss を求めてアームLoRAへ reinforce）  
  3. 【パターンC】傾斜配置された本の検知と直接把持（定常把持、reinforce なし）  
  4. 【パターンD】球体の中心点把持（ビジョンから推定した幾何重心との位置偏差 $loss を reinforce）  
  5. 【パターンC】指定された目標座標への滑らかなアーム誘導（定常走行、reinforce なし）

#### **ROB-03: 複数部品の順次組み立てとステータス更新**

* **概要**: ターゲットリストのパーツ群を順次組み立て、完了ステータスを更新していく。組み立て時の嵌合（かんごう）荷重の最適化やクリアランス精度の調整のため、接合トルクの微細偏差を $loss にして reinforce を行う。  
* **5つの展開バリエーション**:  
  1. 【パターンC】2つのトイブロックの結合タスク（定常実行、reinforce なし）  
  2. 【パターンD】シャフトへのギア部品の挿入（嵌合時の挿入力・摩擦抵抗を $loss にして組み立てLoRAへ reinforce）  
  3. 【パターンC】回路基板へのコネクタ接続（定常実行、reinforce なし）  
  4. 【パターンD】外装カバーへの複数ボルトネジ締め（各ボルトの規定トルク目標値とのばらつき誤差 $loss を reinforce）  
  5. 【パターンC】完成製品へのシリアルラベル正常貼付（定常実行、reinforce なし）

#### **ROB-04: 障害物の自動回避と最適パス移動**

* **概要**: 事前のスキャンで最適パスを計算（静的障害物はスキャンするが例外には至らない）し、計画したテンソルに沿って滑らかに移動。移動エネルギー効率の最大化やJerk（躍度：加速度の変化率）を最小化する軌道パラメータの微調整のため、実行ログから軌道滑らかさの評価誤差を $loss として reinforce する。  
* **5つの展開バリエーション**:

  2. 【パターンD】狭路に置かれた段ボールの回避（旋回GおよびJerk最小化偏差 $loss を算出し経路生成LoRAに reinforce）

  4. 【パターンD】自動ドアのデッドゾーンを避けた通過（測域データから壁への最至近距離マージン誤差 $loss を reinforce）  
  5. 【パターンC】スロープを登坂しながらの傾斜回避移動（定常走行、reinforce なし）

#### **ROB-05: 2台のロボットによる同調協調搬送**

* **概要**: 2台のロボット間で同期テンソル・パルスをやり取りしながら大型荷物を連動して搬送。完全同期して持ち上げるが、物理的な引っ張り合いや遅延を解消するため、2機間の同期位相差や負荷荷重の偏差を $loss に変換して reinforce をかける。  
* **5つの展開バリエーション**:  
  1. 【パターンC】大型段ボール箱の同期持ち上げ搬送（定常搬送、reinforce なし）  
  2. 【パターンD】長尺パイプの並行搬送（自機と相方間の高度なアーム同期遅延および引っ張り負荷 $loss を reinforce）  
  3. 【パターンC】重いテーブルのバランス同期移動（定常搬送、reinforce なし）  
  4. 【パターンD】狭い曲がり角での同期旋回（各ロボット車輪の速度同調誤差 $loss を reinforce）  
  5. 【パターンC】傾斜路前での同調減速と目標配置（定常搬送、reinforce なし）

#### **ROB-06: 規定ルートの巡回監視と充電ドック帰還**

* **概要**: チェックポイントを巡回走行し、定常的な充電ドック接続を行う。ドック挿入時のハードウェアコンタクトの物理的位置調整や、バッテリー充電効率の最適化のため、接続コネクタの嵌合アライメント偏差を $loss と定義して reinforce する。  
* **5つの展開バリエーション**:  
  1. 【パターンC】オフィスエリアの複数デスク島巡回（定常巡回、reinforce なし）  
  2. 【パターンC】倉庫内ラック列の直進監視往復（定常巡回、reinforce なし）  
  3. 【パターンD】充電ドックへの自動高精度接続（ドックコネクタのガイドピン接触力およびアライメントズレ $loss を reinforce）  
  4. 【パターンC】巡回完了後の定常低電力待機状態への移行（定常処理、reinforce なし）  
  5. 【パターンD】カメラ首振りを伴うエリア定点監視（死角を最小化する監視視野アライメント誤差 $loss を reinforce）

### **4\. エラー系（構文）：13シチュエーション (SYN-01 〜 SYN-13)**

#### **SYN-01: 未定義変数の参照エラーからの自己修復**

* **概要**: $balls などの定義されていない変数を参照してエラーとなり、\[EXCEPTION\] を受けて定義を追加してリトライする。  
* **5つの展開バリエーション**:  
  1. 名前の参照漏れ（$name の未定義）  
  2. 数量の参照漏れ（$count の未定義）  
  3. リストの参照漏れ（$items の未定義）  
  4. 設定データの参照漏れ（$config の未定義）  
  5. テンソルの参照漏れ（$t1 の未定義）

#### **SYN-02: ゼロ除算（Division by Zero）の検出と修復**

* **概要**: $x / 0 のようなゼロ除算が発生し、\[EXCEPTION\] を受けて分母を 1 もしくは代替値に修正する。  
* **5つの展開バリエーション**:  
  1. 平均値計算時の分母（データ数）が 0  
  2. 割引率計算時の元の価格が 0  
  3. 等分計算時の人数が 0  
  4. 進行度計算時の全体タスク数が 0  
  5. 物理計算時の質量が 0

#### **SYN-03: 型の不一致（TypeError）の検出とキャストによる修復**

* **概要**: str 型と int 型をそのまま足算しようとしてエラーになり、\[EXCEPTION\] を受けて int キャストを追加する。  
* **5つの展開バリエーション**:  
  1. "3" (文字列) と 2 (整数) の足し算  
  2. "1.5" (文字列) と 0.5 (小数) の足し算  
  3. 変数代入時に int を忘れて str のまま連結演算した場合の修復  
  4. list のインデックスに "0" (文字列) を指定したエラー  
  5. dict のキーにオブジェクトをそのまま指定したエラー

#### **SYN-04: クォーテーション閉じ忘れ（Syntax Error）の修復**

* **概要**: 文字列の末尾クォーテーション " が欠落しエラー。\[EXCEPTION\] の後、正しく閉じ直す。  
* **5つの展開バリエーション**:  
  1. 単紙な変数代入文字列の閉じ忘れ  
  2. amend 内の置換対象文字列 of 閉じ忘れ  
  3. show 引数内の文字列の閉じ忘れ  
  4. ロボットインポート時のモジュール名の閉じ忘れ  
  5. リスト要素内の文字列の閉じ忘れ

#### **SYN-05: リストのインデックス範囲外アクセス（IndexError）の修復**

* **概要**: 要素数3 of リストに対して $l.3 (4番目) にアクセスしてしまいエラー。\[EXCEPTION\] を受けて $l.2 に修正するかリストサイズを確認する。  
* **5つの展開バリエーション**:  
  1. 3つの要素から3番目（インデックス3）を誤って指定  
  2. 空のリストに対してインデックス0にアクセス  
  3. ループや連続アクセスで境界値を超過  
  4. $l.(1+2) など数式評価の結果が範囲外  
  5. リストが削除（del）された後にアクセス

#### **SYN-06: 存在しない dict キーへのアクセス（KeyError）の修復**

* **概要**: dict に存在しないキー $d.unknown を参照しエラー。\[EXCEPTION\] を受けてキーの追加、または正しいキーに修正。  
* **5つの展開バリエーション**:  
  1. user の email キーをスペルミス（emial）  
  2. product の price を参照しようとして cost と書いてしまった場合  
  3. ネストされた dict の中間キーが存在しない  
  4. キー削除（del）した直後にそのキーを参照  
  5. デフォルトキーをセットし忘れた

#### **SYN-07: 無効な関数・演算子の使用エラーの修復**

* **概要**: 実装されていない関数（例: sqrt() の誤用やスペルミス）を呼び出しエラー。\[EXCEPTION\] を経て正しい関数名に修正。  
* **5つの展開バリエーション**:  
  1. cosine() と誤記（正しくは cos()）  
  2. tangent() と誤記（正しくは tan()）  
  3. square() と誤記（正しくは掛け算か対応関数）  
  4. 大文字で SIN() と記述したエラー  
  5. show を print と誤記したエラー

#### **SYN-08: カンマやコロンの欠落（構文エラー）の修復**

* **概要**: dict 定義時（例: a:1 b:2）にカンマが欠落、あるいは amend で \-\> が欠落してエラー。\[EXCEPTION\] を経て修正。  
* **5つの展開バリエーション**:  
  1. dict 宣言時の要素間のカンマ欠落  
  2. dict のキー・値の区切り（コロン）欠落  
  3. amend 文における \-\> のスペルミス（- のみ等）  
  4. list 宣言時の要素間のカンマ欠落  
  5. 関数の引数区切りカンマの欠落

#### **SYN-09: 括弧の不整合（Mismatched Parentheses）の修復**

* **概要**: cos(0 や list(1, 2 などの括弧閉じ忘れエラー。\[EXCEPTION\] を受けて正しく閉じる。  
* **5つの展開バリエーション**:  
  1. 三角関数の括弧閉じ忘れ  
  2. インデックス評価用括弧 .(1 \+ 1 の不整合  
  3. 複雑な算術演算での括弧の開き・閉じミスマッチ

  5. バックスラッシュでエスケープされた括弧と、通常の括弧の混同

#### **SYN-10: 予約語の誤用エラーの修復**

* **概要**: 予約語（str, int, del, return など）を変数名（$str \= 1）として使おうとしてエラー。\[EXCEPTION\] を受けて変数名を修正。  
* **5つの展開バリエーション**:  
  1. 変数名に del を使用  
  2. 変数名に return を使用  
  3. 変数名に int を使用  
  4. 変数名に dict を使用  
  5. 変数名に show を使用

#### **SYN-11: amendの無効な置換ターゲットエラーの修復**

* **概要**: amend で存在しないテキストや、すでに整合性の取れなくなった部分を置換しようとしてエラー。\[EXCEPTION\] を受けて置換対象を正しく再指定。  
* **5つの展開バリエーション**:  
  1. $STATEMENT に含まれていないスペルミス単語の置換試行  
  2. 変更済みの変数に対して、変更前の古い値をターゲットに置換試行  
  3. dict 型そのものに対して文字列 amend を適用しようとしたエラー  
  4. 変数指定 of コロン（:）を忘れた amend  
  5. 空文字列を置換ターゲットに指定してしまったエラー

#### **SYN-12: import対象モジュールのスペルミスと修復（またはraiseによる異常終了）**

* **概要**: ロボット用のライブラリを import する際、存在しない名前（例: robot → robot1）を指定してエラー。\[EXCEPTION\] から修正する。回復不能なインポート失敗時は raise にフォールバックする。  
* **5つの展開バリエーション**:  
  1. import robot（数字の欠落、正しくは robot1 へ自己修復）  
  2. import tensors（複数形ミス、正しくは tensor へ自己修復）  
  3. import sensor1（未定義のセンサーモジュールを検出し、修復を試みる）  
  4. import のスペルミス（impor などを修復）  
  5. **【raise適用】** 存在しないモジュールのロード。リカバリ手段がなく、最終的に raise Error: "Required module not found" を送出し異常終了する。

#### **SYN-13: 修復不能エラー（raiseによる異常終了）**

* **概要**: エラーを検出した後、修復不能であると判断する。\[COMMENT] で回復不能の理由を説明する。\[CODE]raise "..." により処理を終了する。
* **5つの展開バリエーション**:  
  1. 必須設定ファイルが存在しない
  2. 必須データベースが利用できない
  3. ロボット制御サブシステムが初期化できない
  4. 必須 tensor 復元機能が存在しない
  5. 暗号鍵が失われており安全に処理できない

### **5\. エラー系（物理）：12シチュエーション (PHYS-01 〜 PHYS-12)**

物理エラー系では、動作中に発生する各種の例外（\[EXCEPTION\]）に対し、ルーターによるエラー判定（determine）プロセスを経由します。  
リカバリタスクの性質に応じて、以下の2つのアプローチをバリエーション展開で使い分けます。

* **パターンA（シンプル意思決定）**: ルーターで方策を決定（determine）してリカバリを実行し、そのまま正常に完了（reinforce なし）。  
* **パターンB（フィードバック適応型）**: リカバリ結果から得られる誤差テンソル（$loss）を取得し、方策改善のためルーターに学習フィードバック（reinforce）を行う。

#### **PHYS-01: ロボットのつまずき（Stumbled）と歩行リカバリ**

* **概要**: robot1\_walk 中につまずきを検知。エラールーターがバランス回復や歩行再開の方策を determine() し、状況に応じたリカバリを実行。動的バランス回復など一部のケースでは $loss を取得してルーターに reinforce() を適用する。  
* **5つの展開バリエーション**:  
  1. 【パターンA】平地での小石によるつまずき（シンプルなバランス復帰、reinforce なし）  
  2. 【パターンB】階段昇降時のつまずき（高度バランス制御を実行後、ズレ誤差 $loss を算出して reinforce）  
  3. 【パターンA】カーペット端への引っかかり（スライドステップによる回避、reinforce なし）  
  4. 【パターンB】荷物積載時の前傾つまずき（重心モーメント $loss に基づいて reinforce）  
  5. 【パターンA】暗闇での突発的な段差接触（減速停止を伴う安全な立ち直り、reinforce なし）

#### **PHYS-02: 物体の把持失敗（Grasp Failed）と再アプローチ**

* **概要**: マニピュレータの把持失敗を検知。ルーターが、再アプローチか、角度の修正か、締め付けトルク（LoRA）の変更かを determine() し動作。把持失敗時のスリップ度や圧力誤差を $loss としてルーターにフィードバック可能とする。  
* **5つの展開バリエーション**:  
  1. 【パターンA】滑りやすいペットボトルの把持失敗（トルク型LoRAへの切り替え、reinforce なし）  
  2. 【パターンB】紙コップの変形検知による把持失敗（弾性抵抗 $loss を算出し reinforce を行い適応）  
  3. 【パターンA】配置位置ズレによる空振り（ビジョンスキャンを再起動、reinforce なし）  
  4. 【パターンB】重量物の把持滑り（接触センサーの滑り量 $loss をルーターに reinforce）  
  5. 【パターンA】柔らかいぬいぐるみ等の変形を伴う把持（低反発モードへの変更、reinforce なし）

#### **PHYS-03: 介護ロボットの「持ち上げ」時のバランス不整合**

* **概要**: 被介護者を持ち上げた瞬間の急激なトルク・重心変動エラー。ルーターが「下半身姿勢支持」か「アーム吊り上げアシスト」かを determine() する。姿勢の不安定度を $loss に変換し reinforce する強化学習パターンと、単純姿勢補正で済むパターン。  
* **5つの展開バリエーション**:  
  1. 【パターンB】持ち上げ開始時の被介護者の急な動き（重心不整合度 $loss をルーターに reinforce）  
  2. 【パターンA】移乗時の車椅子との角度ミスマッチ（アーム角度の直接補正、reinforce なし）  
  3. 【パターンB】想定より荷重が重い場合の引き上げ姿勢崩れ（足裏圧変化 $loss を reinforce）  
  4. 【パターンA】中腰限界角度の超過（ロボット自体の安全接地・保持姿勢へのシフト、reinforce なし）  
  5. 【パターンB】濡れた床での持ち上げバランス確保（スリップ摩擦 $loss を reinforce）

#### **PHYS-04: センサーの遮蔽（Camera Blocked）と代替センサーへの切り替え**

* **概要**: 主要光学カメラの遮蔽・汚れ例外。ルーターが「LiDARフォールバック」「レンズクリーニング」「触覚ナビ」のいずれを起動すべきか determine()。センサー切り替えなどの決定（パターンA）が主体だが、認識精度フィードバック（パターンB）も組み合わせる。  
* **5つの展開バリエーション**:  
  1. 【パターンA】結露・結氷によるビジョン不全（LiDARと赤外線へのセンサー選択切り替え、reinforce なし）  
  2. 【パターンA】突然の逆光による白飛び（自動シャッター・感度切り替え、reinforce なし）  
  3. 【パターンA】物理的な泥汚れ（エアブローまたはレンズクリーナーモジュールを実行、reinforce なし）  
  4. 【パターンB】物体密着によるカメラ塞がり（触覚フィードバックからの距離ズレ $loss を reinforce）  
  5. 【パターンA】照度低下による夜間暗所モードへの遷移（IRカメラ・アクティブ赤外線照明起動、reinforce なし）

#### **PHYS-05: 移動ルート上の障害物進入（Collision Warning）**

* **概要**: 進路に突発的な動的障害物が進入。ルーターが「急制動停止」「減速＋迂回」「一時待機」などを determine()。急停止はパターンA、移動滑らかさの評価フィードバックはパターンBとする。  
* **5つの展開バリエーション**:  
  1. 【パターンA】前方に歩行者が飛び出してきた（緊急急停止ブレーキLoRAの即時適用、reinforce なし）  
  2. 【パターンB】通路中央への置き荷物の出現（最適経路ズレ $loss を算出してルーターに reinforce）  
  3. 【パターンA】ペットのまとわりつきによる足止め（障害物が離れるまで待機＋チャイム鳴動、reinforce なし）  
  4. 【パターンB】狭い曲がり角での対向ロボット（スムーズな退避軌道の曲率偏差 $loss を reinforce）  
  5. 【パターンA】自動シャッター閉塞による進行不能（代替ゲートへの経路探索リプラン、reinforce なし）

#### **PHYS-06: バッテリー残量警告によるタスク中断と充電ドック帰還（致命的エラー時はraise）**

* **概要**: 動作中にバッテリーレベルが閾値を下回る、または電圧異常を検知。ルーターがタスクの安全な一時停止（パターンA）、充電ドックへの最適帰還、あるいは raise による即時緊急停止かを determine() する。  
* **5つの展開バリエーション**:  
  1. 【パターンA】通常掃除中の残量15%低下（現在マップ位置を退避し、定常帰還、reinforce なし）  
  2. 【パターンA】荷物搬送中のバッテリー不足（荷物を安全な荷下ろしスペースに置いてから帰還、reinforce なし）  
  3. 【パターンA】巡回監視中にドック空き待ちが発生（中継待機スポットで低電力スリープ、reinforce なし）  
  4. 【パターンB】登坂路における高負荷消耗（実消費電力予測値との誤差 $loss を reinforce）  
  5. **【raise適用】** 走行中の電圧低下・セル破断。即時シャットダウンを決定し、raise Error: "Critical Voltage Drop. System shutdown immediately." を送出。

#### **PHYS-07: アームの過負荷・関節の限界（Joint Limit/Overload）**

* **概要**: マニピュレータアームが関節限界（特異点）に到達、またはモーターの駆動過負荷が発生。ルーターが「関節角度の再配置（Inverse Kinematics）」「外力逃がし」などを determine()。軌道の逸脱誤差 $loss を reinforce するなどして関節可動性を改善する。  
* **5つの展開バリエーション**:  
  1. 【パターンA】高所作業時のアーム伸長限界（車輪による体躯前進アシストへ逃げる、reinforce なし）  
  2. 【パターンB】狭所配管へのアーム進入時の干渉（接触圧 $loss をフィードバックして reinforce）  
  3. 【パターンB】ねじ込みトルク限界の超過（関節にかかる過大トルク抵抗 $loss を reinforce）  
  4. 【パターンA】重量ワーク持ち上げ時のモーター加熱（一時関節固定＋放熱ファン駆動、reinforce なし）  
  5. 【パターンB】関節特異点への衝突（位置追従性偏差 $loss をルーターに reinforce）

#### **PHYS-08: 床面の摩擦変化（Slip/Low Traction）の検知とトルク制御**

* **概要**: 駆動輪または歩行脚のスリップ検知例外。ルーターが「スリップ抑制トラクション制御」「多重関節ロック」「４駆切替」を determine()。スリップ比や摩擦推定値を $loss としてルーターに reinforce する。  
* **5つの展開バリエーション**:  
  1. 【パターンB】水濡れ床面でのタイヤスリップ（回転速度差 $loss を算出して reinforce、グリップ力を適応学習）  
  2. 【パターンA】オイル付着床（スリップ率が閾値を超えたため極低速の歩行パターンへ直接切り替え、reinforce なし）  
  3. 【パターンB】傾斜のある砂利道の登坂（車体姿勢の滑り角 $loss を reinforce）  
  4. 【パターンA】泥濘への車輪沈み込み（サスペンションの車高調整と全輪ロック、reinforce なし）  
  5. 【パターンB】凍結路面への進入（微小トラクション偏差 $loss をルーターに reinforce）

#### **PHYS-09: 物体仕分け時の認識曖昧性（Ambiguous Object）**

* **概要**: 画像分類の確信度が低く、仕分けの確実性が落ちるエラー。ルーターが「触覚センサーによる硬度テスト」「多角度撮影」「人間へのクエリ送信」を determine() する。  
* **5つの展開バリエーション**:  
  1. 【パターンA】トイブロックの色判定の曖昧さ（補助照明をONにして再撮影・再判定、reinforce なし）  
  2. 【パターンB】果物とプラスチック模造品の識別（触覚圧力応答 $loss を取得してルーターに reinforce）  
  3. 【パターンA】重なり合う複数の金属ネジのセグメンテーション不全（ロボットアームで優しくかき分けてから再走査、reinforce なし）  
  4. 【パターンB】歪んだ外箱のバーコード読み取りエラー（焦点不整合 $loss を reinforce）  
  5. 【パターンA】認識失敗による未知物体のトレイ除外（安全な一時仕分け先へ直接排出、reinforce なし）

#### **PHYS-10: ロボットハンドの滑り落ち（Object Slipping）**

* **概要**: 把持した対象が自重により下方に滑り始める触覚アラート。ルーターが「即時追加握力」「両手抱え込み」「落下の緩和着地」などを determine()。滑り加速度を $loss に換算して reinforce を行い、最適保持力をオンライン更新する。  
* **5つの展開バリエーション**:  
  1. 【パターンB】細身のガラス器具の滑り落ち検知（静止摩擦からの位置変動 $loss を reinforce し把持圧を適応）  
  2. 【パターンA】段ボール表面の滑り（瞬時把持圧LoRAをロードしトルク増大、reinforce なし）  
  3. 【パターンB】油圧シリンダーシャフトの保持漏れ（剪断応力変化 $loss を reinforce）  
  4. 【パターンA】完全に滑り落ちたため衝突回避（床面接触前に安全退避姿勢を直接指令、reinforce なし）  
  5. 【パターンB】薄膜プラスチックバッグの引きちぎれ限界（引き裂き力上限 $loss を reinforce）

#### **PHYS-11: 環境の急激な変化（強風・振動・地震など、不可避な場合はraise）**

* **概要**: 屋外突風や屋内の床振動などの環境ノイズ例外。ルーターが「姿勢高低重心化」「作業一時中断」「構造安全退避」を determine()。制御可能ならパターンAまたはB、物理的限界に達した場合は raise を呼び出す。  
* **5つの展開バリエーション**:  
  1. 【パターンA】強風環境下での荷役ドローン（風上への姿勢傾斜制限の適用、reinforce なし）  
  2. 【パターンB】突然の大きな床振動（高周波変位スペクトル $loss をルーターに reinforce）  
  3. 【パターンA】降雨による防水レベル超過の懸念（退避用シェルターへの自律走行退避、reinforce なし）  
  4. **【raise適用】** 突風による車体転倒。復旧不可能なため、raise Error: "Overturned. Hardware damage critical." でシステム遮断。  
  5. 【パターンB】精密工場での高床微振動（微細変位 $loss を reinforce して微小防振アーム制御）

#### **PHYS-12: 複数協調作業時の他ロボットのトラブル検知（通信・物理不可避時はraise）**

* **概要**: ペアで連動しているロボットから通信遮断またはエラー報告例外。ルーターが「自機の単独姿勢保持への移行（パターンA）」「同調パラメータリセット」、あるいは協調崩壊時の安全な raise を determine()。

  5. 【パターンB】双腕ロボットの左右同期ズレ（軸整合トルク $loss を reinforce）
